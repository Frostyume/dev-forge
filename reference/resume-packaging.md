# resume —— 把复盘锻造成简历项目

`/dev-forge resume [zh|en] [--jd <文件或文本>] [--anon]`
默认 `zh`。执行前先读 `conventions.md`。

## 1. 目标与产物

把已解决问题归纳成 STAR 简历项目，维护一份**活文档**，可随时增量更新：

- `resume/resume-project.zh.md` / `resume/resume-project.en.md`（主稿，按语言）
- `resume/resume-project.<lang>.<jdslug>.md`（针对某 JD 的裁剪变体，可选）

## 2. 选材

扫 `problems/`，**只取 `status: resolved`** 的问题作为素材，并优先级排序：

- 优先：`severity` 高、`影响`明确、`根因`有技术深度、`解决方案`体现能力的问题。
- 弱化/合并：low 且琐碎的问题不单独成条，可并入"此外修复若干环境/依赖问题"。
- 按 `project` 字段**分组**，每个 project 产出一个简历项目块（含 STAR 贡献 + 单行 bullet）。

## 3. STAR 写法规范

- 每条贡献写全 **S/T 背景 → A 你的动作与方法 → R 可量化结果**。
- 多用动词开头（定位/重构/优化/修复/设计/搭建）。
- 量化优先：耗时下降、稳定性/性能提升、影响范围、复发归零等；记录里有 `time_spent`、severity、impact 可直接用。
- 体现"根因深度"——简历价值不在"修了 bug"，而在"定位到非显然的根因并系统性解决"。

## 4. 更新流程（增量、保留手工润色）

1. 读现有主稿（若存在）及其 frontmatter `source_problems`。
2. 扫 `problems/` 取当前所有 resolved 的 id 集合。
3. 求差：
   - **新增** resolved（不在 source_problems）→ 起草新 bullet，插入对应 project 块。
   - **已移除/改回未解决** → 标注待复核，不擅自删用户已润色的句子。
4. 文档分两区：`## 关键贡献` 为可再生区；约定一个 `## 手工润色（保留）` 区，更新时**原样保留**，绝不覆盖用户改过的措辞。
5. 更新 `updated` 与 `source_problems`，回执：新增/变更了哪些条目。

## 5. 语言

`zh` / `en` 各自独立主稿；用户要双语就分别生成。不做机器对译，按各自语言习惯重写。

## 6. JD 裁剪（可选）

给 `--jd`（文件路径或粘贴文本）时：

- 抽取 JD 关键词（技能栈、职责动词、领域）。
- 从主稿中**重排/精选**最匹配的 bullet，调整措辞贴合 JD 用语，置顶高匹配项。
- 写到 `resume/resume-project.<lang>.<jdslug>.md`，**不改主稿**。jdslug 取 JD 公司/岗位的短 slug。

## 7. 脱敏（可选）

给 `--anon` 时：

- 将真实 `repo`/`project`/内部代号替换为通用描述（如"某 EDA 布局系统"、"某分布式训练框架"）。
- commit 哈希、内部路径、敏感数字一并隐去或泛化。
- 映射关系只留在本地、不写进产物。

## 8. 真实性红线

**只写记录支持得起的内容**。无量化数据时用真实的定性表述，不得编造指标、不得夸大角色。简历是对外材料，宁可朴素也不失真。
